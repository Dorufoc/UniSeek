# UniSeek 全平台业务逻辑设计 V2

> 本文档以文字形式逐一叙述 UniSeek 兼职招聘平台每个功能的详细业务逻辑，涵盖前端交互、后端处理、数据流转、状态变更和约束规则，供前后端开发人员对照实现。

---

## 一、用户注册功能

用户注册是系统的入口功能，支持手机号和邮箱作为双唯一标识。用户在注册页面填写手机号（11 位数字，以 1 开头）、邮箱（标准邮箱格式）、密码（6 到 20 位）和昵称，同时需要在身份选择单选按钮中选择注册身份：求职者（`role=0`）或企业HR（`role=1`）。前端在提交前需要对手机号格式（11 位纯数字，1 开头）、邮箱格式（包含 @ 符号的标准邮箱正则）和密码长度（6 到 20 位）进行基础校验，同时确保身份已选择且值只能为 0 或 1。根据身份选择的不同，注册成功后前端跳转的引导页面也有所不同：求职者跳转至简历编辑页，企业HR 跳转至企业资质提交页。

后端接到注册请求后，第一步校验 `role` 参数只能为 0（求职者）或 1（企业HR），禁止传入 9（普通管理员）或 99（超级管理员），管理员账号只能由超级管理员在后台设置。第二步查询 `user` 表确认该手机号是否已被注册，如果 `uk_phone` 唯一索引冲突则返回「该手机号已注册」的提示。第三步查询该邮箱是否已被注册，如果 `uk_email` 唯一索引冲突则返回「该邮箱已被注册」的提示。第四步，使用 `SecureRandom` 生成 16 字节随机数，转换为 32 位十六进制字符串作为该用户的独立盐值（`salt`）。第五步，将用户输入的明文密码与盐值拼接后，进行 MD5 加密，得到 32 位密文存入 `password` 字段。采用独立盐值的目的是即便两个用户设置了相同的密码，在数据库中存储的密文也不相同，增强安全性。第六步，向 `user` 表插入一条新记录，`role` 字段设为用户选择的身份值（0 或 1），`status` 默认设为 1（正常状态），`credit_score` 默认设为 100。第七步，生成 JWT Token，将用户 ID 和角色编码进 Token 的 payload 中，设置有效期 30 分钟，将 Token 返回给前端。

前端收到注册成功的响应后，将 Token 存储在本地（localStorage 或 cookie 中），并根据 `role` 值跳转到不同页面：求职者跳转到职位列表首页，企业HR 跳转到企业资质提交页面引导其完成企业认证。

## 二、用户登录功能

登录页面提供手机号和密码输入框。用户输入手机号和密码后提交，前端对手机号格式做基础校验。

后端接到登录请求后，第一步根据手机号查询 `user` 表，如果查不到记录则返回「手机号或密码错误」（不明确提示哪一项错误，防止撞库攻击）。第二步检查 `status` 字段，如果为 0（禁用状态），则返回「账号已被禁用，请联系管理员」。第三步，从查询结果中取出该用户注册时生成的独立盐值（`salt`），将用户输入的密码与盐值拼接后进行 MD5 加密，将结果与数据库中的 `password` 字段比对。如果不匹配则返回「手机号或密码错误」。第四步，登录成功，更新 `last_login_time` 字段为当前时间。第五步，生成 JWT Token，payload 中编码用户 ID 和角色值（0 求职者 / 1 企业HR / 9 普通管理员 / 99 超级管理员），设置 30 分钟有效期，将 Token 返回给前端。

前端收到登录成功的响应后，保存 Token，解析 Token 中的角色信息，根据角色值跳转到不同的首页视图。求职者看到职位列表首页，企业HR 看到企业管理后台首页，普通管理员和超级管理员均看到运营管理后台首页。

系统内置一个超级管理员种子账号，手机号为 `18688886666`，密码为 `admin`，`role=99`。该账号作为数据库种子数据在系统初始化时插入 `user` 表，密码同样经过 MD5+盐加密存储，登录流程与普通用户完全一致，走手机号扫库校验。超级管理员账号不可被禁用或删除。

## 三、JWT 鉴权与权限控制功能

系统通过 JWT Token 实现无状态鉴权。后端使用 `JwtInterceptor` 拦截所有 HTTP 请求，在 `WebConfig` 中注册拦截器，配置白名单路径（如 `/api/auth/register`、`/api/auth/login` 等）直接放行，不需要 Token。

对于需要鉴权的请求，拦截器从请求头的 `Authorization: Bearer <token>` 中提取 Token，使用配置的 `jwt.secret` 密钥验证 Token 的签名是否有效，同时检查 Token 是否已过期（30 分钟有效期）。如果 Token 无效或过期，返回 401 Unauthorized 错误码。

Token 校验通过后，拦截器从 Token payload 中解析出用户 ID 和角色值，然后根据请求 URL 的路径前缀进行角色权限匹配。`/api/admin/*` 路径下的请求允许角色值大于等于 9 的用户访问（即 `role=9` 的普通管理员和 `role=99` 的超级管理员均有权限），如果角色不匹配返回 403 Forbidden。`/api/enterprise/*` 路径下的请求仅允许角色值为 1（企业HR）的用户访问。`/api/resume/*` 路径下的请求仅允许角色值为 0（求职者）的用户访问。其他通用接口（如职位列表浏览）不做角色限制，但需要用户已登录。

权限校验通过后，拦截器将用户 ID 和角色信息存入 `ThreadLocal`，供后续的 Controller 和 Service 层直接获取当前用户信息，无需从请求参数中解析。

## 四、实名认证功能

实名认证是求职者投递职位和 HR 提交企业资质的前置条件。用户进入实名认证页面或弹窗后，填写真实姓名和身份证号（18 位）。

后端接到实名认证请求后，第一步使用 Hutool 工具类的 `IdcardUtil.isValidCard()` 方法校验身份证号的合法性，校验规则包括长度必须为 18 位，前 17 位为数字，最后一位为数字或 X，前 6 位地址码合法，第 7 到 14 位出生日期格式正确且日期真实存在，最后一位校验码通过前 17 位数字与加权因子相乘求和后模 11 映射得到。第二步使用 `IdcardUtil.getAgeByIdCard()` 从身份证号中提取出生日期并计算年龄，如果不满 16 周岁则返回「未满 16 周岁，暂不可使用平台服务」的提示。第三步校验通过后，查询 `real_name_auth` 表，如果该用户已有认证记录（`uk_user_id` 唯一索引），则更新记录；如果没有则插入新记录。身份证号在入库前需要进行加密处理（如 AES 加密），保障用户隐私安全。第四步将 `status` 置为 1（已认证），记录 `auth_time` 为当前时间。第五步返回认证成功结果。

实名认证是一次性操作，一旦认证通过，后续不再提示。前端在用户进入投递流程或企业资质提交流程时，先调用 `GET /api/auth/real-name/status` 查询认证状态，如果未认证则弹出实名认证弹窗引导用户完成认证，认证通过后继续原流程。

## 五、企业资质提交功能

企业资质提交是 HR 用户的核心功能，HR 注册后必须先提交企业资质并通过运营审核，才能发布职位。

HR 进入企业资质提交页面时，前端首先调用 `GET /api/auth/real-name/status` 接口检查实名认证状态。如果未实名认证，弹出实名认证弹窗，引导用户完成实名认证后再继续。实名认证通过后，显示资质填写表单，包含公司全称（必填，不超过 100 字）、统一社会信用代码（必填，18 位字母数字组合）、营业执照图片（通过文件上传组件上传，后端处理后返回图片 URL）、所属行业（从预定义行业列表中选择）、公司简介（选填，不超过 2000 字）。

前端在表单提交前执行基础校验：公司全称不能为空，信用代码必须匹配 18 位字母数字正则，营业执照图片必须已上传。校验通过后调用 `POST /api/enterprise` 接口。

后端接到请求后执行以下校验链：第一步从 JWT 中获取当前用户 ID，查询 `user` 表确认角色为 1（HR），如果不是 HR 则拒绝。第二步查询 `real_name_auth` 表确认已实名认证（`status = 1`），如果未认证返回特定错误码。第三步查询 `enterprise` 表确认该用户尚未创建过企业记录（`uk_user_id` 唯一约束）。第四步校验信用代码格式（18 位字母数字），查询 `enterprise` 表确认该信用代码未被其他企业使用（`uk_credit_code` 唯一约束）。第五步所有校验通过后，将企业信息插入 `enterprise` 表，`audit_status` 设置为 0（待审状态），`create_time` 和 `update_time` 设置为当前时间。

插入成功后返回企业记录 ID 和当前状态。前端收到响应后跳转到资质审核进度页面，显示「已提交，等待运营审核」的状态标签。

## 六、企业资质状态查询功能

HR 提交资质后，可以随时在企业中心查看审核进度。前端调用 `GET /api/enterprise/my` 接口，后端从 JWT 中获取当前用户 ID，查询 `enterprise` 表返回该用户的企业记录。

前端根据返回的 `audit_status` 值展示不同的 UI。`audit_status = 0`（待审）时，显示「审核中」标签，发布职位按钮置灰，提示「企业资质审核中，请耐心等待」。`audit_status = 1`（已认证）时，显示「已认证」绿色标签，发布职位按钮可用。`audit_status = 2`（驳回）时，显示「已驳回」红色标签，展示驳回原因，显示「重新提交」按钮引导用户修改后重新提交。

## 七、企业资质修改功能（驳回后重新提交）

当企业资质被管理员驳回后，HR 需要修改信息后重新提交。前端在检测到 `audit_status = 2` 时，加载已提交的信息回填到表单中，并在表单顶部展示驳回原因，帮助用户针对性修改。

用户在表单中修改相关字段（可以修改公司全称、信用代码、重新上传营业执照等），完成后调用 `PUT /api/enterprise` 接口。后端接到请求后，第一步校验当前用户的企业记录 `audit_status` 确实是 2（驳回状态），如果是 1（已认证）则不允许修改。第二步如果信用代码被修改，校验新的信用代码是否与其他企业冲突。第三步更新数据库中的字段，同时将 `audit_status` 重置为 0（待审状态），`update_time` 更新为当前时间。

重置为待审状态后，该企业重新进入运营管理员的待审队列，需要重新经过审核流程。

## 八、管理员审核企业资质功能

运营管理员（包括普通管理员 `role=9` 和超级管理员 `role=99`）登录后台后，进入企业资质审核页面。页面默认展示所有待审核（`audit_status = 0`）的企业列表，按提交时间倒序排列，方便管理员优先处理最新申请。管理员也可以通过下拉筛选查看已认证或已驳回的记录列表。

管理员点击某条记录进入详情页，查看企业提交的完整信息，包括公司全称、信用代码、营业执照图片（可点击放大查看）、所属行业、公司简介等。管理员根据这些信息的真实性和完整性做出审核决定。

如果审核通过，管理员点击「通过」按钮，调用 `PUT /api/admin/enterprise/audit/{id}`，请求体传入 `audit_status = 1`。如果审核驳回，管理员填写驳回原因后点击「驳回」，请求体传入 `audit_status = 2` 和 `reject_reason`（驳回原因）。

后端接到审核请求后，第一步校验当前操作人角色是否大于等于 9（`role >= 9`）。第二步查询待审核记录是否存在且 `audit_status` 当前为 0（防止重复审核）。第三步更新 `audit_status` 字段和 `update_time`。第四步调用通知服务，向该企业的 HR 用户发送审核结果通知。审核通过时通知内容为「您的企业资质已通过审核，可以发布职位了」，审核驳回时通知内容为「您的企业资质审核未通过，原因：[驳回原因]，请修改后重新提交」。

前端操作成功后刷新列表，当前记录从待审列表中移除，状态变更生效。

## 九、简历创建与编辑功能

求职者注册后可以创建和管理自己的在线简历。每个求职者只有一份简历，与用户 1:1 绑定（`resume.user_id` 唯一索引）。

求职者进入简历编辑页面，填写以下信息：性别（男/女）、出生日期、学历（如本科、硕士、大专等，从预定义列表中选择）、毕业院校、技能标签（以逗号分隔或 JSON 数组格式存储）、工作或实践经历（富文本编辑器，支持格式排版）、附件简历上传（支持 PDF 和 Word 格式文件）。

前端在提交前校验必填字段（如学历不能为空），校验通过后调用 `PUT /api/resume` 接口（使用 upsert 方式，有记录则更新，无记录则插入）。后端接到请求后，第一步从 JWT 中获取当前用户 ID，校验角色为求职者（`role = 0`）。第二步查询 `resume` 表是否存在该用户的简历记录，如果存在则更新，如果不存在则插入新记录。第三步将处理结果返回给前端。

简历的 `experience` 字段设置了全文索引（FULLTEXT），支持 HR 在后端通过关键词搜索简历内容，例如搜索「Java 开发」可以找到经历中包含该关键词的求职者。

## 十、简历附件上传功能

求职者可以在编辑简历时上传附件简历文件（PDF 或 Word 格式）。前端使用文件上传组件，限制文件类型为 pdf、doc、docx，文件大小一般不超过 10MB。用户选择文件后调用 `POST /api/resume/attachment` 接口上传。

后端接到上传请求后，校验文件类型和大小，将文件存储到配置的文件存储目录（本地磁盘或云存储），生成访问 URL，将该 URL 更新到 `resume` 表的 `attachment_url` 字段中，返回 URL 给前端。前端将 URL 展示为下载链接，方便 HR 查看简历详情时下载附件。

## 十一、职位分类浏览功能

系统内置了 30 条职位分类种子数据，采用两级树形结构（15 个顶级分类和 15 个子级分类）。顶级分类包括餐饮服务、家教辅导、快递物流、促销导购、话务客服、设计创作、文案写作、技术支持、翻译校对、美容美发、家政保洁、教育培训、活动策划、摄影摄像、其他等。每个顶级分类下有若干子分类，例如餐饮服务下有服务员、后厨帮工、送餐员、收银员、咖啡师/调酒师。

前端在职位发布表单和职位浏览页面中需要展示分类选择器。分类数据通过 `GET /api/category/tree` 接口加载，后端查询 `parent_id IS NULL` 的记录获取顶级分类，按 `sort_order` 排序，然后对每个顶级分类递归查询 `parent_id` 等于该分类 ID 的子级分类，组装成树形 JSON 返回给前端。

前端使用级联选择器展示两级分类，用户先选顶级分类（如餐饮服务），再选子级分类（如服务员）。分类数据变化频率低，前端可以缓存较长时间。

## 十二、行政区划浏览功能

系统内置了全国 3432 条行政区划种子数据（GB/T 2260 标准），分为省（level=1，34 条）、市（level=2，342 条）、区县（level=3，3056 条）三级树形结构。

前端在职位发布表单和职位搜索页面中需要展示地区级联选择器。加载策略是三级渐进加载：第一步加载省级列表（`GET /api/region/provinces`），第二步用户选择省份后加载该省下辖的市级列表（`GET /api/region/children/{parentId}`），第三步用户选择城市后加载该市下辖的区县级列表。

`region` 表使用 `idx_level_parent` 复合索引（`level, parent_id`）加速级联查询，同时 `parent_id` 自关联外键设置为 `SET NULL`，确保删除上级行政区划时不会影响已关联的职位数据。

## 十三、职位发布功能

企业 HR 的企业资质通过审核后（`audit_status = 1`），就可以发布兼职职位了。

HR 点击「发布职位」进入职位发布表单，需要填写以下信息：职位标题（如「奶茶店周末兼职服务员」，不超过 100 字）、职位描述（富文本编辑器，包含工作内容、任职要求等）、薪资范围下限和上限（`salary_min` 和 `salary_max`，业务约束 `salary_min ≤ salary_max`）、薪资单位（日结/时薪/月结，单选）、岗位类型（全职/兼职/实习）、招聘总人数（`total_quota`）、报名截止时间（`deadline`）、工作地址文本描述、工作地区（从地区级联选择器中选择）、职位分类（从分类级联选择器中选择）。

前端在提交前校验表单必填项和业务约束（如薪资下限不能大于上限，招聘人数必须大于 0），校验通过后调用 `POST /api/task/publish` 接口。

后端接到发布请求后，第一步从 JWT 中获取当前用户 ID，校验角色是否为 HR（`role = 1`）。第二步查询 `enterprise` 表获取该 HR 关联的企业信息，校验 `audit_status` 是否为 1（已认证），如果企业未认证则返回「企业资质未认证，暂不可发布职位」。第三步将职位信息插入 `task` 表，`status` 默认设置为 0（待审状态），`remaining_quota` 等于 `total_quota`（初始时全部名额可用），`version` 乐观锁版本号设置为 0。第四步记录操作日志（`operation_type = TASK_PUBLISH`）。

插入成功后，职位进入待审队列，等待运营管理员审核上架。

## 十四、职位审核功能

运营管理员（包括普通管理员 `role=9` 和超级管理员 `role=99`）在后台的职位审核页面查看所有待审核（`status = 0`）的职位列表。列表展示职位标题、发布企业、薪资范围、发布时间等信息，按提交时间倒序排列。

管理员点击职位进入详情页，查看职位完整信息，包括标题、描述、薪资、分类、地区、招聘名额等。管理员根据内容的真实性和合规性做出审核决定。

审核通过时，调用 `PUT /api/admin/task/audit/{id}`，`status` 更新为 1（招聘中）。审核驳回时，填写驳回原因，`status` 保持不变（0-待审），驳回原因通过通知发送给 HR，HR 修改后可重新提交审核。

职位一旦审核通过（`status = 1`），就在前端职位列表中可见，求职者可以浏览和投递。

## 十五、职位列表浏览与搜索功能

求职者进入平台首页或职位列表页面，可以按多种维度浏览和搜索招聘中的职位。列表默认展示所有 `status = 1`（招聘中）的职位，按发布时间倒序排列，并支持分页加载。

前端提供以下筛选条件：按职位分类筛选（两级树形选择器）、按工作地区筛选（省/市/区三级级联选择器）、按薪资范围筛选（最低薪资和最高薪资的滑动条或输入框）、按岗位类型筛选（全职/兼职/实习多选）、按关键词搜索职位标题和描述。

后端根据筛选条件动态构建 SQL 查询，使用 `idx_status_create` 复合索引（`status, create_time`）加速按状态加时间排序的查询，使用 `idx_category_status` 复合索引加速分类加状态的筛选，使用 `idx_region_status` 复合索引加速地区加状态的筛选。所有查询都限制 `status = 1`（招聘中），确保求职者看不到已满员、已过期或已下架的职位。

返回结果包括职位的基本信息、企业信息（公司名称、行业）、以及剩余名额（`remaining_quota`）用于前端展示「还剩 X 个名额」的提示。

## 十六、职位状态自动变更功能

职位在招聘过程中会根据业务条件自动变更状态。系统设计了两条自动变更路径。

第一条是名额归零自动满员。当 HR 录用求职者时，会扣减 `task.remaining_quota` 字段。如果扣减后 `remaining_quota = 0`，系统立即将 `status` 自动更新为 2（已满员），该职位不再接受新的投递。

第二条是截止时间自动过期。系统通过 Spring `@Scheduled` 定时任务，在每日凌晨 01:00 执行一次批量更新，将所有 `status = 1`（招聘中）且 `deadline < NOW()`（已超过报名截止时间）的职位自动标记为 `status = 3`（已过期）。

此外，HR 也可以在招聘过程中手动下架职位。HR 点击「下架」按钮，调用 `PUT /api/task/offline/{id}`，后端将 `status` 更新为 4（已下架）。下架后的职位不再出现在前端列表中，已投递的求职者记录不受影响。

## 十七、职位乐观锁防超录功能

为了防止多个 HR 同时录用求职者导致实际录用人数超过招聘总人数（超录），`task` 表设计了乐观锁版本号字段 `version`。

当 HR 录用求职者时，后端首先执行 `SELECT remaining_quota, version FROM task WHERE id = ? AND status = 1` 查询当前剩余名额和版本号。如果 `remaining_quota <= 0`，返回「名额不足」的错误提示。如果名额充足，执行 UPDATE 操作：`UPDATE task SET remaining_quota = remaining_quota - 1, version = version + 1 WHERE id = ? AND version = 旧版本号`。

这个 UPDATE 语句的 WHERE 条件中包含了 `version = 旧版本号`，意味着如果有两个事务同时读取到了同一个版本号并都想执行扣减，第一个事务的 UPDATE 会成功（affected rows = 1），第二个事务的 UPDATE 会因为版本号已被更新而失败（affected rows = 0）。当 affected rows = 0 时，表示发生了乐观锁冲突，后端需要重试整个流程或返回「名额不足」的错误提示。

扣减完成后，系统检查更新后的 `remaining_quota` 是否等于 0，如果等于 0 则自动将 `status` 更新为 2（已满员）。

## 十八、投递职位功能

求职者浏览职位列表时，找到感兴趣的职位后点击「投递」按钮进行报名。

前端在点击投递时，首先调用 `GET /api/auth/real-name/status` 检查实名认证状态。如果未认证，弹出实名认证弹窗，引导用户完成认证后才能继续投递。如果已认证，检查职位状态：`status` 必须为 1（招聘中）、职位未过期（`deadline > NOW()`）、剩余名额大于 0。如果职位不可投递，提示相应原因。

校验通过后，前端调用 `POST /api/application/deliver` 接口。后端接到请求后，第一步从 JWT 获取求职者 ID，校验角色为求职者（`role = 0`）。第二步再次校验实名认证状态。第三步查询 `task` 表确认职位可投递（`status = 1`、`deadline > NOW()`、`remaining_quota > 0`）。第四步查询 `task_application` 表的 `uk_task_applicant` 唯一索引（`task_id, applicant_id`），确认该求职者没有投递过同一职位，如果已投递则返回「已投递过该职位」的提示。第五步读取 `resume` 表的完整简历信息，将所有字段序列化为 JSON 快照（`resume_snapshot`），这个快照记录了投递时刻求职者的简历状态，投递后求职者修改简历不会影响已经投递的记录。第六步向 `task_application` 表插入一条新记录，`status` 设置为 0（已投递），`version` 设置为 0。第七步自动创建一条 `chat_session` 记录，关联该投递记录 ID，`employer_id` 为发布职位的企业 HR 用户 ID，`seeker_id` 为求职者用户 ID，`status` 设置为 0（活跃状态）。第八步向 HR 发送系统通知，通知内容为「有新的投递」，附带投递记录 ID 供点击跳转。

前端收到投递成功响应后，跳转到投递记录页面，显示「投递成功」的状态标签，同时可以在消息中心看到与 HR 的聊天会话已自动创建。

## 十九、简历快照机制

简历快照是投递功能的重要设计。当求职者点击投递时，系统读取 `resume` 表的当前完整数据（包括性别、出生日期、学历、学校、技能、经历、附件 URL 等），将所有字段序列化为一个 JSON 字符串，作为 `resume_snapshot` 字段存储在 `task_application` 表中。

这个快照机制的核心目的是保证投递记录的客观真实性。如果投递后求职者修改了自己的在线简历，HR 在审核投递时看到的仍然是投递时的简历状态，而不是修改后的最新版本。这防止了求职者在投递后「事后修改简历」导致的争议。

从数据流来看，`resume` 表始终存储简历的最新版本，`task_application.resume_snapshot` 存储每个投递时刻的历史快照。HR 在简历池中查看投递时，读取 `resume_snapshot` 字段展示给 HR，而不是从 `resume` 表实时读取。

## 二十、投递状态管理功能（HR 侧）

HR 进入简历池页面，可以查看某个职位下的所有投递记录列表。列表按投递时间倒序排列，每条记录显示求职者的简历快照信息、投递时间和当前状态。

HR 对每条投递记录进行审核处理，状态变更遵循严格的流转规则，不可跳跃、不可回退。初始状态为 0（已投递），HR 可以将其变更为以下三种状态之一：1（待面试）表示邀请求职者面试，HR 需要填写面试时间和面试地点，系统自动向求职者发送面试邀请通知；2（待定）表示暂不决定，留待后续处理；4（已淘汰）表示该求职者不符合要求，HR 需要填写淘汰原因（必填），系统自动向求职者发送淘汰通知。

对于状态为 1（待面试）的记录，HR 在面试结束后可以将其变更为以下三种状态之一：3（已录用）表示面试通过，此时系统执行乐观锁名额扣减，扣减成功后向求职者发送录用通知；2（待定）表示面试后暂不确定，留待后续决定；4（已淘汰）表示面试未通过，填写淘汰原因后发送淘汰通知。

对于状态为 2（待定）的记录，HR 后续可以将其变更为 1（重新安排面试）、3（录用）或 4（淘汰），操作逻辑同上。

状态 3（已录用）是终态前的状态，后续只能变更为 5（已完成），表示求职者已实际到岗工作并结算完成。状态 4（已淘汰）和 5（已完成）是最终状态，一旦到达不可再变更为其他任何状态。

每次状态变更，系统都会记录操作日志（`operation_type` 分别为 `APPLICATION_HIRE`、`APPLICATION_REJECT` 等），并通过 `notification` 表向求职者发送相应类型的通知（面试邀请通知 `type = 1`、录用通知 `type = 2`、淘汰通知 `type = 3`）。

## 二十一、投递状态查看功能（求职者侧）

求职者投递职位后，可以在「我的投递」页面查看所有投递记录及其当前状态。列表按投递时间倒序排列，每条记录显示职位标题、企业名称、投递时间和当前状态标签。

前端根据不同的 `status` 值展示不同的状态标签和操作按钮：0（已投递）显示「已投递，等待 HR 处理」灰色标签；1（待面试）显示「待面试」蓝色标签，并展示 HR 填写的面试时间和面试地点；2（待定）显示「待定」黄色标签，提示「HR 暂未做出决定」；3（已录用）显示「已录用」绿色标签，提示「恭喜您被录用！」；4（已淘汰）显示「已淘汰」红色标签，并展示 HR 填写的淘汰原因；5（已完成）显示「已完成」灰色标签，表示工作已结束。

## 二十二、消息通知功能

消息通知系统在关键业务节点自动向用户推送站内信。通知记录存储在 `notification` 表中，核心字段包括接收人 ID、发送人 ID（NULL 表示系统自动通知）、标题、内容、类型、已读状态和关联业务 ID。

触发通知的业务场景共有四个：第一，求职者投递成功后，系统向 HR 发送一条类型为 0（系统通知）的消息，标题为「新投递提醒」，内容包含求职者昵称和职位名称，`biz_id` 为投递记录 ID。第二，HR 邀请面试时，向求职者发送一条类型为 1（面试邀请）的消息，标题为「面试邀请」，内容包含面试时间和地点，`biz_id` 为投递记录 ID。第三，HR 录用求职者时，向求职者发送一条类型为 2（录用通知）的消息。第四，HR 淘汰求职者时，向求职者发送一条类型为 3（淘汰通知）的消息，内容包含淘汰原因。

用户可以在消息中心查看自己的所有通知列表，列表按时间倒序排列，分页加载。未读消息数量在导航栏以红点形式展示，用户可以点击标记单条已读或一键全部已读。

## 二十三、即时聊天功能

求职者投递成功后，系统自动创建一个聊天会话（`chat_session`），关联该投递记录。一个投递记录对应一个会话（1:1 关系），HR 和求职者通过这个会话进行点对点沟通。

用户进入消息中心时，加载自己的会话列表。HR 查询 `employer_id = 当前用户 ID` 的活跃会话，求职者查询 `seeker_id = 当前用户 ID` 的活跃会话。列表按 `last_message_time` 倒序排列，展示对方昵称、最后一条消息摘要和未读消息数。会话列表的查询直接使用 `chat_session` 表的 `last_message` 和 `last_message_time` 字段，无需聚合 `chat_message` 表，大幅提升加载性能。

用户点击某个会话进入聊天页面，加载最近 20 条消息的历史记录。聊天消息存储在 `chat_message` 表中，使用 `idx_session_time` 复合索引（`session_id, send_time`）支持高效的分页查询。用户上拉可以滚动加载更多历史消息（每次 20 条）。

用户输入文本内容点击发送，或选择图片发送，前端调用 `POST /api/chat/message/send` 接口。后端接到请求后，第一步校验发送方是否属于该会话（发送方必须是该会话的 `employer_id` 或 `seeker_id`）。第二步校验消息内容非空且不超过 2000 字。第三步向 `chat_message` 表插入消息记录，`is_read` 设为 0（未读），`send_time` 设为当前时间。第四步更新 `chat_session` 表的 `last_message` 为消息内容摘要，`last_message_time` 为当前时间。第五步返回发送成功结果，前端将消息实时展示在聊天界面中。

当接收方进入聊天页面时，前端调用接口将该会话中所有发送方为对方且 `is_read = 0` 的消息批量标记为已读（`is_read = 1`），同时更新会话列表中的未读数量。

## 二十四、投诉举报功能

用户在使用平台过程中，如果遇到企业发布虚假信息或其他用户的不当行为，可以发起投诉举报。

投诉人（求职者或 HR）进入投诉页面，选择被投诉对象的类型（1-企业、2-用户，注意平台设计中被投诉对象仅针对企业和个人用户，不对具体职位进行投诉），输入被投诉对象的 ID 或从相关页面跳转时自动带入，选择投诉类型（如虚假信息、违规行为、其他等，`type` 字段存储投诉类型编码），填写投诉内容（详细描述），然后提交。

后端接到投诉请求后，向 `complaint` 表插入记录，`status` 默认设置为 0（待处理状态），记录投诉人 ID、被投诉对象类型和 ID、投诉类型、投诉内容和创建时间。`target_type` 字段的值为 1（企业）或 2（用户），根据前端选择的被投诉对象类型确定。

投诉提交后，运营管理员（包括普通管理员 `role=9` 和超级管理员 `role=99`）在后台的投诉管理页面看到待处理的投诉列表。管理员点击某条投诉查看详情，然后点击「认领」将该投诉分配给自己处理，`status` 更新为 1（处理中），`handler_id` 更新为管理员 ID。管理员调查后，如果认定投诉属实，填写处理结果（如对违规企业进行下架处理、对违规用户进行禁用处理等），然后点击「结案」，`status` 更新为 2（已结案），系统向投诉人发送处理结果通知。如果认定投诉不属实，管理员填写驳回原因后结案。

## 二十五、运营日报统计功能

系统通过 Spring `@Scheduled` 定时任务，在每日凌晨 00:05 自动统计前一天的平台运营数据，写入 `daily_statistics` 表。

定时任务的执行逻辑如下：第一步确定统计日期为昨天（`LocalDate.now().minusDays(1)`）。第二步查询 `daily_statistics` 表中是否已有该日期的记录（`uk_stat_date` 唯一索引），如果有则跳过（幂等保护，防止定时任务重复执行导致重复统计）。第三步如果没有记录，并行执行 7 个 COUNT 查询：统计昨天新增用户数（`COUNT user WHERE create_time` 属于昨天）、昨天新增认证企业数（`COUNT enterprise WHERE audit_status = 1 AND update_time` 属于昨天）、昨天新增职位数（`COUNT task WHERE create_time` 属于昨天）、昨天新增简历数（`COUNT resume WHERE create_time` 属于昨天）、昨天新增投递数（`COUNT task_application WHERE create_time` 属于昨天）、昨天新增面试数（`COUNT task_application WHERE status = 1 AND update_time` 属于昨天）、昨天新增入职数（`COUNT task_application WHERE status = 3 AND update_time` 属于昨天）。第四步将统计结果汇总后插入 `daily_statistics` 表。

管理员在运营看板中可以查看日报统计，按日期范围筛选，查看平台运营趋势。统计数据每日一条，支持折线图展示长期走势。

## 二十六、操作日志审计功能

系统对所有关键业务操作进行自动审计日志记录。日志写入 `operation_log` 表，该表设计为只追加、不修改、不删除的审计存储。

需要记录日志的操作类型包括：用户注册（`USER_REGISTER`）、用户登录成功（`USER_LOGIN`）、企业资质提交（`ENTERPRISE_SUBMIT`）、管理员审核企业资质（`ENTERPRISE_AUDIT`）、HR 发布职位（`TASK_PUBLISH`）、管理员审核职位（`TASK_AUDIT`）、职位下架（`TASK_OFFLINE`）、求职者投递（`APPLICATION_DELIVER`）、HR 录用求职者（`APPLICATION_HIRE`）、HR 淘汰求职者（`APPLICATION_REJECT`）、管理员处理投诉（`COMPLAINT_HANDLE`）、用户完成实名认证（`REAL_NAME_AUTH`）、超级管理员设置管理员（`ADMIN_SET_ROLE`）。

日志记录的内容包括操作人 ID（如果是系统自动操作则为 NULL）、操作类型编码、操作目标类型（表名）、操作目标 ID、操作详情（JSON 格式，记录变更前后的关键字段值）、操作 IP 地址（从 `HttpServletRequest.getRemoteAddr()` 获取，兼容 IPv6）。

日志的记录方式有两种：一种是通过 AOP 切面拦截关键方法自动记录，另一种是在 Service 层的关键业务方法中显式调用日志服务写入。

日志仅限运营管理员通过 `GET /api/admin/logs` 接口查询，支持按操作人、操作类型、时间范围筛选。日志数据不可修改、不可删除，用于安全审计和问题追溯。

## 二十七、管理员后台用户管理功能

运营管理员（包括普通管理员 `role=9` 和超级管理员 `role=99`）在后台可以查看平台所有用户的列表，支持按角色（求职者/企业HR/普通管理员/超级管理员）筛选、按注册时间排序、按手机号或昵称搜索。

管理员点击用户详情查看用户的完整信息，包括注册时间、最后登录时间、角色、实名认证状态、信用积分等。对于违规用户，管理员可以将其账号禁用（`status` 设置为 0），禁用后该用户无法登录平台。对于已禁用的用户，管理员也可以将其恢复为正常状态（`status` 设置为 1）。

用户禁用操作需要谨慎，禁用后该用户的所有关联数据（简历、投递记录、聊天记录等）仍然保留在数据库中，只是登录被拦截。

此外，超级管理员（`role=99`）在用户管理页面可以看到「设为管理员」和「取消管理员」按钮。点击「设为管理员」可将目标用户的 `role` 从 0 或 1 修改为 9（普通管理员）；点击「取消管理员」可将目标用户的 `role` 从 9 修改回 0（求职者）。普通管理员（`role=9`）看不到这两个按钮，无权修改其他用户的角色。超级管理员不可对自己的账号执行「取消管理员」操作，平台至少保留一个超级管理员账号。操作成功后系统记录操作日志（`ADMIN_SET_ROLE`），包含变更前后的角色值。

## 二十八、管理员后台职位管理功能

运营管理员（包括普通管理员 `role=9` 和超级管理员 `role=99`）在后台可以查看所有职位列表，包括待审核、招聘中、已满员、已过期、已下架等各种状态。管理员可以按状态、企业、分类、地区等维度筛选。

对于已上架的职位，如果发现内容违规或企业存在问题，管理员可以直接将该职位下架（`status` 更新为 4），同时记录操作原因。下架后职位不再对求职者可见，已投递的记录不受影响。

## 二十九、管理员后台统计看板功能

运营管理员（包括普通管理员 `role=9` 和超级管理员 `role=99`）进入统计看板页面，查看平台的运营概况。看板分为总览数据和趋势数据两部分。

总览数据显示平台的累计数据：累计注册用户数、累计认证企业数、累计发布职位数、累计投递次数等。这些数据通过聚合查询各业务表获得。

趋势数据展示每日新增用户数、新增企业数、新增职位数、新增投递数、新增面试数、新增入职数的折线图。数据来源于 `daily_statistics` 表，管理员可以选择日期范围查看短期或长期趋势。

## 三十、超级管理员账号管理功能

系统内置一个超级管理员种子账号，作为数据库初始化种子数据插入 `user` 表。该账号的具体信息如下：手机号为 `18688886666`，密码为 `admin`（经 MD5+随机盐加密后存储），`role` 设为 99，`status` 设为 1（正常），`nickname` 设为「超级管理员」或「系统管理员」。该账号与其他用户共用同一张 `user` 表，登录流程完全一致，不设任何特殊通道。

超级管理员拥有与普通管理员完全相同的后台权限，可以执行所有 `/api/admin/*` 路径下的操作，包括企业资质审核、职位审核、投诉处理、数据统计查看、操作日志查询等。两者的 API 权限无差别，拦截器对 `role >= 9` 的用户一视同仁放行。

超级管理员额外拥有一项特殊权限：在用户管理页面中可以将其他用户设置为普通管理员（`role=9`），也可以将普通管理员降级为普通用户（`role=0`）。此项操作的判断依据是当前登录用户的 `role` 是否为 99，前端根据当前用户的角色值决定是否渲染「设为管理员」和「取消管理员」按钮，后端在接口中校验当前操作人的 `role` 是否为 99。

安全约束方面，超级管理员账号 `18688886666` 不可被禁用（`status` 不可为 0），不可被删除，不可被降级。平台上至少保证一个超级管理员账号存在。超级管理员的操作同样会被记录到操作日志审计表中，便于追溯。
