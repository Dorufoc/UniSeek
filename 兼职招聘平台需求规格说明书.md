## 1. 项目背景与目标

### 1.1 项目背景

随着数字经济的发展，兼职招聘市场呈现出快速增长态势。然而，现有招聘平台普遍存在以下痛点：

- **信息真实性不足**：虚假职位信息泛滥，求职者权益难以保障
- **投递流程混乱**：求职者重复投递、HR 简历管理低效
- **数据缺乏规范**：兼职岗位的薪资结算、状态流转缺乏标准化机制
- **用户体验不佳**：注册流程繁琐（依赖短信验证码），平台操作流程冗长

基于上述背景，优寻（UniSeek）招聘平台应运而生。

### 1.2 项目目标

构建一个 **轻量、高效、可信赖** 的兼职招聘平台，实现以下目标：

1. 提供 **手机号+密码** 的极简注册/登录体验，无需第三方短信服务
2. 实现 **职位发布-审核-上架-满员** 的全生命周期管理
3. 简历投递采用 **快照机制**，确保投递记录的客观真实性
4. 严格 **名额控制与状态流转**，防止超录与流程混乱
5. 采用 **乐观锁** 机制保障并发安全
6. 系统采用模块化设计，为未来微服务拆分奠定基础

### 1.3 适用范围

本需求规格说明书适用于以下阶段的研发与测试工作：

- 系统架构设计与技术选型
- 数据库物理表结构设计
- 前后端功能开发
- 接口联调与集成测试
- 系统验收与交付

#### 3.1.5 实名认证

##### 功能概述

在求职者投递职位、HR 提交企业资质时，系统弹出实名认证弹窗，要求用户完成身份证实名认证。认证成功后永久有效，同一账号不再重复提示。

##### 认证规则

1. **触发场景**：
   - 求职者点击"立即投递"时，若未实名则弹出认证窗口，认证通过后方可继续投递
   - HR 提交企业资质时，若未实名则弹出认证窗口，认证通过后方可提交
2. **一次性认证**：认证成功后 `real_name_auth.status = 1`，后续不再弹出
3. **认证方式**：用户输入真实姓名 + 身份证号，系统后端校验

##### 身份证号校验逻辑

采用开源工具库 Hutool 的 `IdcardUtil` 进行校验，校验流程如下：

1. **非空校验**：姓名和身份证号均不能为空
2. **格式校验**（`IdcardUtil.isValidCard`）：
   - 18 位长度，前 17 位为数字，最后 1 位为数字或大写字母 X
   - 地址码（前 6 位）合法
   - 出生日期（第 7~14 位）格式正确、日期真实存在
3. **校验码计算**（`IdcardUtil.isValidCard18` 内部包含）：
   - 前 17 位每位数字乘以对应加权因子 `[7, 9, 10, 5, 8, 4, 2, 1, 6, 3, 7, 9, 10, 5, 8, 4, 2]`
   - 求和后对 11 取模，余数映射到校验字符 `['1', '0', 'X', '9', '8', '7', '6', '5', '4', '3', '2']`
   - 映射结果与身份证最后一位比对，一致则通过
4. **年龄校验**：实名认证时年龄需 ≥ 16 周岁

##### 后端处理流程

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  前端     │         │  后端    │         │  Hutool  │
└────┬─────┘         └────┬─────┘         └──────────┘
     │ POST /api/auth/    │                    │
     │ real-name          │                    │
     │ (realName, idCard) │                    │
     │───────────────────→│                    │
     │                    │ IdcardUtil.        │
     │                    │ isValidCard(idCard)│
     │                    │───────────────────→│
     │                    │←───────────────────│
     │                    │  false → 身份证号   │
     │                    │  格式不正确         │
     │                    │                    │
     │                    │ IdcardUtil.        │
     │                    │ getAgeByIdCard(    │
     │                    │  idCard)           │
     │                    │  < 16 → 未满        │
     │                    │  16周岁            │
     │                    │                    │
     │                    │ 校验通过 → 写入      │
     │                    │ real_name_auth   │
     │                    │ status = 1         │
     │                    │                    │
     │ {success: true}    │                    │
     │←───────────────────│                    │
     │                    │                    │
```

##### 核心校验代码

```java
import cn.hutool.core.util.IdcardUtil;

public class RealNameAuthService {

    /**
     * 实名认证校验
     * @param realName 真实姓名（非空）
     * @param idCard   身份证号（18位）
     * @return 校验结果
     */
    public boolean verify(String realName, String idCard) {
        // 1. 非空校验
        if (realName == null || realName.trim().isEmpty()) {
            throw new IllegalArgumentException("姓名不能为空");
        }
        if (idCard == null || idCard.trim().isEmpty()) {
            throw new IllegalArgumentException("身份证号不能为空");
        }

        // 2. 身份证格式 + 校验码校验（Hutool 内部完成）
        if (!IdcardUtil.isValidCard(idCard)) {
            throw new IllegalArgumentException("身份证号格式不正确");
        }

        // 3. 年龄校验（16周岁以上）
        int age = IdcardUtil.getAgeByIdCard(idCard);
        if (age < 16) {
            throw new IllegalArgumentException("未满16周岁，无法完成实名认证");
        }

        return true;
    }
}
```

| 基本流程 | 1. HR 首次使用需提交企业资质<br>2. 系统校验该用户是否已完成实名认证<br>   - 若未认证 → 弹出实名认证弹窗（触发 UC-AUTH-05）<br>   - 认证完成后继续后续流程<br>3. 填写公司全称、统一社会信用代码、上传营业执照图片<br>4. 选择企业所在地区<br>5. 选择所属行业，填写公司简介<br>6. 提交后资质状态为"待审核"<br>7. 运营管理员审核通过后，状态变更为"已认证"<br>8. 审核驳回时 HR 可查看原因并重新提交 |

**UC-EMPLOYER-02：发布职位**

| 项目 | 内容 |
|---|---|
| 参与者 | 企业 HR（企业资质已认证） |

**UC-EMPLOYER-04：名额控制与录用**

| 项目 | 内容 |
|---|---|
| 参与者 | 企业 HR |
| 核心规则 | 1. 发布职位时填写"招聘总人数"，系统初始化"剩余名额"= 招聘总人数<br>2. HR 每录用一名求职者，系统扣减剩余名额 1<br>3. 扣减操作使用乐观锁（`version` 字段），防止多名 HR 同时录用导致超录<br>4. 剩余名额降至 0 时，职位状态自动变更为 2（已满员）<br>5. 已满员的职位不可继续录用 |

#### 3.5.4 聊天室模块

##### 功能概述

提供 HR 与求职者之间的点对点私聊功能。当求职者对某个职位投递简历后，HR 与求职者可基于该投递记录开启一对一聊天会话。聊天支持文本消息发送、消息已读/未读状态标识、历史消息加载与消息实时推送。

##### 核心规则

1. **会话建立条件**：求职者投递职位后，HR 与求职者之间自动建立聊天会话关系，无需手动添加好友
2. **消息隔离**：每条消息绑定到具体的投递记录（`task_application_id`），不同投递的消息互不干扰
3. **权限控制**：
   - HR 仅能查看与自家企业职位的求职者的聊天记录
   - 求职者仅能查看与自身投递相关的聊天记录
   - 管理员可查看所有聊天记录（用于投诉仲裁）
4. **消息存储**：所有聊天记录持久化存储，支持历史消息翻页加载

##### 用例描述

**UC-CHAT-01：发送聊天消息**

| 项目 | 内容 |
|---|---|
| 参与者 | 求职者、企业 HR |
| 前置条件 | 求职者已对该职位投递简历，存在对应的 `task_application` 记录 |
| 后置条件 | 消息写入 `chat_message` 表，接收方可在聊天列表中看到新消息 |
| 基本流程 | 1. 用户进入聊天会话页面<br>2. 输入消息内容（文本）并点击发送<br>3. 系统校验发送方与目标投递记录的关联性（HR 需是该职位的发布企业成员，求职者需是该投递的申请人）<br>4. 系统插入 `chat_message` 记录，`is_read` 默认为 0（未读）<br>5. 返回发送成功，消息实时展示在聊天界面 |
| 异常流程 | 权限校验失败 → 返回"无权发送消息"<br>消息内容为空或超长（>2000字）→ 返回相应提示 |

**UC-CHAT-02：查看聊天会话列表**

| 项目 | 内容 |
|---|---|
| 参与者 | 求职者、企业 HR |
| 基本流程 | 1. 用户进入"消息中心"页面<br>2. 系统展示该用户的所有聊天会话列表<br>3. 每个会话项展示：对方头像/昵称、关联职位标题、最新一条消息内容及时间<br>4. 未读消息的会话列表高亮显示（加粗/标记）<br>5. 列表按最新消息时间倒序排列 |

**UC-CHAT-03：加载聊天历史消息**

| 项目 | 内容 |
|---|---|
| 参与者 | 求职者、企业 HR |
| 前置条件 | 用户已进入某个聊天会话 |
| 基本流程 | 1. 进入聊天会话时，加载最近 20 条消息<br>2. 用户上拉/点击"加载更多"时，加载更早的历史消息（分页加载，每页 20 条）<br>3. 系统自动将对方发送的未读消息标记为已读（`is_read = 1`）<br>4. 消息按发送时间正序排列，最新消息在底部 |

## 4. 非功能性需求

### 4.1 并发安全

| 需求 | 说明 |
|---|---|
| 超录防护 | 多名 HR 同时录用同一职位的候选人时，使用乐观锁（`version` 字段）保证剩余名额不会被扣减为负数。更新时校验 `version = oldVersion`，失败则重试或提示"名额不足"。 |

### 4.2 响应时间

| 场景 | 目标 |
|---|---|
| 职位列表页加载 | ≤ 2 秒 |
| 职位详情页加载 | ≤ 1.5 秒 |
| 登录/注册接口 | ≤ 500ms |
| 投递操作接口 | ≤ 1 秒 |
| 列表类接口（分页） | ≤ 1 秒 |

### 4.3 可用性

| 需求 | 说明 |
|---|---|
| 运行时间 | 系统支持 7×24 小时不间断运行 |
| 定时任务 | 定时任务自动处理过期职位（报名截止时间已过的职位自动变更为"已过期"） |
| 故障恢复 | 数据库定时备份，关键业务操作记录日志 |

### 4.4 扩展性

| 需求 | 说明 |
|---|---|
| 模块化分包 | 代码按功能模块分包（`auth`、`job`、`resume`、`message`、`chat`、`statistics`、`admin`） |
| 微服务演进 | 当前单体架构但模块解耦，未来可按模块拆分为独立微服务 |
| 接口规范 | 前后端通过 RESTful API 交互，数据格式统一为 JSON |

### 4.5 安全性

| 需求 | 说明 |
|---|---|
| 密码存储 | MD5 + 随机盐加密，禁止明文存储 |
| 密码策略 | 长度 6~20 位，支持字母、数字、特殊字符组合 |
| 手机号唯一性 | 全库唯一索引约束 |
| Token 安全 | JWT 携带用户身份信息，敏感接口必须校验 Token |
| 数据隔离 | 企业 HR 仅能查看本企业数据；求职者仅能查看个人数据 |

## 5. 数据字典

### 5.1 表命名规范

- 表名采用小写字母 + 下划线风格（snake_case），无统一前缀
- 字段名采用小写字母 + 下划线风格（snake_case）
- 主键统一命名为 `id`
- 外键字段以 `_id` 结尾
- 乐观锁字段统一命名为 `version`
- 时间字段统一命名为 `create_time` / `update_time`，消息类表使用 `send_time`

### 5.2 数据库表结构

#### 5.2.1 user（用户表）

用户认证核心表，存储所有角色的账号信息。

|---|---|---|---|---|

| avatar_url | VARCHAR | 255 | DEFAULT NULL | 头像URL |

| credit_score | INT | 10 | NOT NULL, DEFAULT 100 | 信用积分 |
| status | TINYINT | 1 | NOT NULL, DEFAULT 1 | 状态：0禁用/1正常 |
| last_login_time | DATETIME | - | DEFAULT NULL | 最后登录时间 |

## 6. 核心业务状态机说明

### 6.1 投递记录状态定义

| 状态值 | 状态名称 | 含义 |
|---|---|---|
| 0 | 已投递 | 求职者已投递简历，等待 HR 处理 |
| 1 | 待面试 | HR 已标记为待面试，已发送面试邀请 |
| 2 | 待定 | HR 暂未做最终决定，处于观望状态 |
| 3 | 已录用 | HR 已录用该求职者 |
| 4 | 已淘汰 | HR 已淘汰该求职者 |
| 5 | 已完成 | 该兼职工作已结算完成 |

### 6.2 状态流转图

```
                         ┌─────────────┐
                         │   已投递(0)  │
                         └──────┬──────┘
                                │
                ┌───────────────┼───────────────┐
                │               │               │
                ▼               ▼               ▼
         ┌──────────┐   ┌──────────┐   ┌──────────┐
         │ 待面试(1) │   │  待定(2)  │   │ 已淘汰(4) │
         └─────┬────┘   └─────┬────┘   └──────────┘
               │              │
               ├──────────────┤
               │              │
               ▼              ▼
         ┌──────────┐   ┌──────────┐
         │ 已录用(3) │   │ 已淘汰(4) │
         └─────┬────┘   └──────────┘
               │
               ▼
         ┌──────────┐
         │ 已完成(5) │
         └──────────┘
```

### 6.3 状态流转约束（严格规则）

| 源状态 | 可转换到的目标状态 | 说明 |
|---|---|---|
| 0（已投递） | 1（待面试）、2（待定）、4（已淘汰） | HR 首次处理时的三种选择 |
| 1（待面试） | 2（待定）、3（已录用）、4（已淘汰） | 面试后的处理 |
| 2（待定） | 1（待面试）、3（已录用）、4（已淘汰） | 待定后可重新安排面试或最终决定 |
| 3（已录用） | 5（已完成） | 录用后完成结算 |
| 4（已淘汰） | -（终态） | 淘汰后不可逆转 |
| 5（已完成） | -（终态） | 完成后不可逆转 |

**核心约束：**

1. **不可跳跃**：例如 0（已投递）不可直接到 3（已录用），必须经过 1（待面试）
2. **不可回退**：状态只能正向流转，到达"已录用""已淘汰""已完成"后不可回退到之前状态
3. **已淘汰不可逆转**：一旦标记为淘汰，不可重新录用
4. **已完成不可逆行**：结算完成后状态锁定

### 6.4 职位状态定义与流转

| 状态值 | 状态名称 | 含义 |
|---|---|---|
| 0 | 待审核 | 企业已提交，等待运营审核 |
| 1 | 招聘中 | 审核通过，求职者可浏览投递 |
| 2 | 已满员 | 剩余名额为 0，自动变更 |
| 3 | 已过期 | 报名截止时间已过，自动变更 |
| 4 | 已下架 | 企业手动下架或运营强制下架 |

**职位状态流转：**

```
                  ┌─────────────┐
          ┌──────→│   待审核(0)  │←──────┐
          │       └──────┬──────┘       │
          │              │ 审核通过      │ 审核驳回
          │              ▼              │
          │       ┌─────────────┐       │
          │       │  招聘中(1)   │       │
          │       └──────┬──────┘       │
          │              │              │
     ┌────┴────┐   ┌────┴────┐   ┌────┴────┐
     │ 已满员(2)│   │ 已过期(3)│   │ 已下架(4)│
     └─────────┘   └─────────┘   └─────────┘
```

### 6.5 状态流转的触发条件与副作用

| 流转路径 | 触发动作 | 附带操作 |
|---|---|---|
| 已投递 → 待面试 | HR 点击"邀请面试" | 创建面试邀请消息（type=1），记录面试时间/地点 |
| 待面试/待定 → 已录用 | HR 点击"录用" | 扣减 `task.remaining_quota`（乐观锁）；若剩馀名额归零则职位自动变更为"已满员"；创建录用通知消息（type=2） |
| 状态 → 已淘汰 | HR 点击"淘汰" | 需填写淘汰原因，创建淘汰通知消息（type=3） |
| 已录用 → 已完成 | 结算确认 | 更新结算金额相关的统计字段 |
| 职位满员 → 已满员 | 自动触发 | 系统在投递列表中对满员职位标注"已满员"，求职者不可投递 |

### 7.3 MD5 随机盐加密实现规范

```java
// 密码加密（注册时调用）
public String encryptPassword(String password, String salt) {
    // 盐值拼接在密码后方
    String source = password + salt;
    // 进行 MD5 摘要（可迭代多次增加强度）
    return DigestUtils.md5DigestAsHex(source.getBytes(StandardCharsets.UTF_8));
}

// 随机盐生成
public String generateSalt() {
    SecureRandom random = new SecureRandom();
    byte[] saltBytes = new byte[16]; // 128位
    random.nextBytes(saltBytes);
    // 转为32位十六进制字符串
    return Hex.encodeHexString(saltBytes);
}

// 密码校验（登录时调用）
public boolean verifyPassword(String inputPassword, String storedPassword, String salt) {
    return storedPassword.equals(encryptPassword(inputPassword, salt));
}
```

### 7.4 JWT Token 规范

```
Header:
{
  "alg": "HS256",
  "typ": "JWT"
}

Payload:
{
  "userId": 1,           // 用户ID
  "role": 0,             // 角色标识
  "phone": "138xxxx",    // 手机号（脱敏）
  "exp": 1688888888,     // 过期时间（当前时间 + 30分钟）
  "iat": 1688887088      // 签发时间
}

签名：
HMAC-SHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secretKey)
```

### 7.5 鉴权拦截器

所有非公开接口（除 `/api/auth/register`、`/api/auth/login` 外）均需携带 JWT Token 并在请求头中传递：

```
Authorization: Bearer <token>
```

**鉴权流程：**

1. 前端在 HTTP 请求头携带 Token
2. 后端 Filter/Interceptor 拦截请求，提取 Token
3. 解析 Token → 验证签名 → 验证有效期 → 提取用户身份
4. 将用户信息存入请求上下文（ThreadLocal）
5. 根据接口的权限注解校验用户角色是否匹配
6. 校验通过 → 放行；校验失败 → 返回 401/403
